# DB Migration

Use this skill when a backend change requires schema evolution, data backfill, or compatibility planning at the database layer.

## Responsibilities

- Translate application-level persistence changes into safe Alembic revisions.
- Keep schema evolution aligned with SQLAlchemy models, repositories, services, and Unit of Work behavior.
- Plan backward compatibility, rollout order, and rollback safety.
- Make data backfills and destructive changes explicit.

## Default Workflow

1. Clarify the use case and why the schema must change.
2. Identify affected tables, columns, constraints, indexes, enums, and data assumptions.
3. Check which application layers must change with the migration: repositories, service logic, DTOs, controllers, or jobs.
4. Decide whether the migration is additive, destructive, or expand-and-contract.
5. Write or review the Alembic revision manually; do not trust autogenerated output blindly.
6. If data needs reshaping, decide whether it belongs inside the revision, in a separate backfill script, or in a staged rollout.
7. Verify upgrade and downgrade paths as far as the change realistically allows.
8. Verify the application still works against the schema at the rollout stage you are targeting.

## Migration Types

- Additive: new nullable columns, new tables, new indexes, new optional relations.
- Contracting: dropping columns, tightening constraints, removing enums or tables.
- Expand-and-contract: dual-write or staged compatibility when old and new shapes must coexist temporarily.

Prefer additive or staged changes when production compatibility matters.

## Core Rules

- Keep migrations deterministic and reviewable.
- Avoid mixing major schema refactors, business logic rewrites, and backfills in one opaque revision.
- Schema changes should not force controllers or services to bypass the existing Unit of Work or repository boundaries.
- Update SQLAlchemy models and repository assumptions together with the migration plan.
- If a migration changes write behavior, verify the affected service still has a valid transaction boundary.

## Alembic Guidance

- Prefer explicit Alembic operations over giant autogenerated diffs.
- Name revisions clearly enough to understand intent later.
- Review autogenerated SQLAlchemy diffs for accidental renames, dropped defaults, or constraint churn.
- Keep downgrade support when it is realistic; if not realistic, document why and what rollback means operationally.

## Backward Compatibility

- Do not deploy destructive schema changes before the application no longer depends on the old shape.
- For renames, prefer staged expand-and-contract over immediate drop-and-recreate.
- For non-null additions, prefer: add nullable column -> backfill -> update app -> enforce constraint.
- For enum or constraint tightening, verify old producers and background jobs will not emit now-invalid values.

## Data Backfill Guidance

- Keep heavy or long-running backfills out of request paths.
- Decide explicitly whether backfill runs:
  - inside the migration revision
  - as a separate script or job
  - in multiple staged batches
- Make idempotency and resumability explicit for non-trivial backfills.
- Log progress and failure points for operationally meaningful backfills.

## Rollout And Rollback

- Prefer rollout order: expand schema -> deploy compatible app -> backfill -> enforce stricter constraints -> remove old schema pieces.
- Define what rollback means:
  - database downgrade
  - app rollback on still-compatible schema
  - operational fallback without immediate downgrade
- Be realistic: some data migrations are not safely reversible once new writes happen.
- If rollback is limited, document the safe fallback path clearly.

## Verification

- Run upgrade locally on a representative database state.
- Run downgrade when supported.
- Verify ORM mappings, repository queries, and write paths against the migrated schema.
- Verify indexes, constraints, and defaults behave as intended.
- Add or update integration tests for critical persistence behavior.

## Red Flags

- Dropping or renaming columns without a staged compatibility plan.
- Combining schema changes with unbounded data rewrites in one risky deployment.
- Migrations that require controllers or services to reach around repositories or UoW.
- Assuming autogenerate output is correct without review.
- Forgetting background jobs, seeds, or old workers that still depend on the previous schema.

## Handoff

Return:
- Alembic revisions added or changed
- schema objects affected
- compatibility and rollout plan
- backfill or downgrade notes
- application layers that must ship with the migration
- tests or manual verification run
